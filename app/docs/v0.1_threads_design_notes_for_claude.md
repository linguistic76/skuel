# v0.1 — Threads in SKUEL.app (Design Notes for Claude Code)

This file is a short, practical guide to help you (Claude Code) think in terms of **threads** while working within SKUEL.app’s existing architecture.

This is not a directive to redesign SKUEL or introduce heavy new abstractions. It’s a framing that can guide small, safe decisions.

---

## 1) What a thread is

A **thread** is a persistent line of meaning that a person returns to over time. It is not a page type and not a database table by default. It is a way to organize experience.

A thread can contain many items (goals, habits, tasks, notes). One item can belong to multiple threads.

Examples of threads in plain language:
- Morning rhythm
- Energy and sleep
- Game of Life: Choices to Habits
- Learning how to handle anxiety

---

## 2) The single principle that protects threads

**Do Not Break Continuity**

Every design decision should preserve a user’s ability to return to what they were already in, without penalty, confusion, or reset.

Operational test:
- If someone leaves for two weeks, can they return without shame and without needing to reconstruct context?

If a change makes the user feel “behind,” “starting over,” or “outside of something they were part of,” it breaks continuity.

---

## 3) Quiet vocabulary (optional, internal)

This is not branding. It is internal language that helps keep the system humane.

- Thread: Continuum
- Resume: Return
- Goal: Marker
- Habit: Rhythm
- Task: Action
- Progress: Deepening
- Inactive: Dormant or Paused
- Dashboard: Ground

---

## 4) Mapping to current SKUEL concepts

These mappings are conceptual; do not introduce new models unless a small prototype proves value.

- Goals are natural **Markers** (finite targets with status and time).
- Habits are natural **Rhythms** (recurring patterns that can weaken or strengthen).
- Tasks are natural **Actions** (small doables serving rhythms and markers).
- Goal “details” content already supports **Deepening** (why, guidance, derivation) rather than pure CRUD.

---

## 5) Where threads fit the current UI architecture

The current inbound adapters are domain-based: /goals, /habits, etc.

A thread-aware approach does not remove domains. It composes across them.

Small design guideline:
- Domains remain the building blocks.
- Threads are composed views that bring relevant domain objects into one place.

---

## 6) Minimal, safe experiment (read-only Threads page)

Goal: introduce the concept of threads as a compositional view, without adding new persistence.

Proposed experiment:
- Add an inbound UI route: GET /threads
- Display a small list of curated threads (hard-coded for now).
- Each thread entry shows:
  - name
  - last touched
  - 1–3 related items (goals/habits/tasks) pulled from existing queries or placeholders

Keep it read-only. No “create thread” UI. No new database table.

Purpose: validate whether the orientation improves clarity and return.

---

## 7) Avoid these thread-breaking patterns

These patterns can be technically correct but psychologically destructive for threads:

- Streak reset to zero as primary signal
- “Overdue” language that implies moral failure
- Completed items disappearing from view by default
- Navigation that forces the user to pick context again (instead of re-entering where they were)
- A single “inbox” that flattens all meaning into a queue

Thread-preserving alternatives:
- “Last return” instead of “missed”
- “Paused” instead of “failed”
- “Dormant” instead of “abandoned”
- “Continuing” section to re-enter ongoing work

---

## 8) Implementation posture

Keep the work grounded:

- Prefer small prototypes over architecture debates.
- Prefer composition over new types.
- Prefer preserving continuity over adding features.
- If uncertain, pause and validate assumptions with a minimal slice.
